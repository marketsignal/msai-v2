# Runbook: IaC Parity Re-Apply (post-Slice-4 merge)

**Purpose:** Re-apply Bicep to prod RG after Slice 4 merges. Lands `vmMiReaderAssignment` (and any other Slice 4 changes) declaratively. Also surfaces any Slice 3 manual patches that were granted manually and need parity.

**When:** Once, right after Slice 4 PR merges to main.
**Estimated time:** 10-15 min.
**Cost:** $0 (no new compute; only role assignment + lifecycle policy + alerts).

---

## 1. Confirm pre-conditions

```bash
az account show --query name -o tsv     # expected: MarketSignal2
az group show -n msaiv2_rg --query name -o tsv   # expected: msaiv2_rg
```

## 2. Get current operator IP + Entra OID

```bash
OPERATOR_IP=$(curl -sf https://ifconfig.me)
OPERATOR_OID=$(az ad signed-in-user show --query id -o tsv)
SSH_PUB=$(cat ~/.ssh/<your-pubkey>.pub)   # same key used for Slice 1
```

## 3. Run what-if first (cheap dry-run)

```bash
az deployment group what-if \
  --name main \
  --resource-group msaiv2_rg \
  --template-file infra/main.bicep \
  --parameters infra/main.bicepparam \
  --parameters "operatorIp=$OPERATOR_IP" \
               "operatorPrincipalId=$OPERATOR_OID" \
               "vmSshPublicKey=$SSH_PUB" \
  --no-pretty-print > /tmp/whatif.json
```

**Inspect carefully:**

```bash
jq '.changes[] | {changeType, resourceId: .resourceId | split("/") | .[-2:] | join("/")}' /tmp/whatif.json
```

Expected output (Slice 4):

- `Create` on `vmMiReaderAssignment` (the manually-granted Reader role, now declarative)
- `Create` on `backupsLifecycle` (storage lifecycle policy)
- `Create` on `msai-ops-alerts`, `msai-app-insights`, `msai-health-ping`, 4 × `scheduledQueryRules` — all NEW Slice 4 resources
- `NoChange` on every existing Slice 1+3 resource

**Red flags — STOP and investigate:**

- Any `Delete` operation on an existing resource
- `Modify` on anything other than `Microsoft.Insights/components` (App Insights ingestion mode toggles can show benign Modify on first apply)
- Drift on Slice 1's NSG (means Slice 3's `ghOidcNsgContributorAssignment` parity isn't holding)

## 4. Apply

```bash
az deployment group create \
  --name main \
  --resource-group msaiv2_rg \
  --template-file infra/main.bicep \
  --parameters infra/main.bicepparam \
  --parameters "operatorIp=$OPERATOR_IP" \
               "operatorPrincipalId=$OPERATOR_OID" \
               "vmSshPublicKey=$SSH_PUB"
```

~5-10 min. Outputs same Slice 1/3 values plus new Slice 4 outputs (`actionGroupId`, `appInsightsId`).

The `module activityLog` reference in `main.bicep` is sub-scoped (deploys
the Activity Log diagnostic setting at SUBSCRIPTION scope). The RG-scoped
`deployment group create` above can nest a sub-scoped module — confirm in
the output that you see "activity-log-<hash>" succeed.

If the deployment fails specifically on `Microsoft.Insights/diagnosticSettings`
with `AuthorizationFailed`, your operator role is RG-scoped (Contributor)
rather than subscription-scoped (Owner / Monitoring Contributor). Two options:

1. Get Monitoring Contributor at subscription scope (preferred — narrow grant)
2. Deploy the activity-log module standalone with sub-scoped what-if first:
   ```bash
   az deployment sub what-if -l eastus2 -f infra/activity-log.bicep \
     -p logAnalyticsWorkspaceId=$(az monitor log-analytics workspace show \
       -g msaiv2_rg -n msai-law-* --query id -o tsv)
   az deployment sub create -l eastus2 -f infra/activity-log.bicep \
     -p logAnalyticsWorkspaceId=$(az monitor log-analytics workspace show \
       -g msaiv2_rg -n msai-law-* --query id -o tsv)
   ```

## 5. Verify the new role assignment

```bash
VM_MI=$(az vm identity show -g msaiv2_rg -n msai-vm --query principalId -o tsv)
az role assignment list \
  --assignee-object-id "$VM_MI" \
  --scope /subscriptions/68067b9b-943f-4461-8cb5-2bc97cbc462d/resourceGroups/msaiv2_rg \
  --query "[?roleDefinitionName=='Reader']" -o table
```

Expected: 1 row, RoleAssignmentName = guid generated by Bicep.

## 6. Manual Slice 3 patches — verify they no-op now

```bash
# 1. Network Contributor on NSG for ghOidcMi
GH_OIDC_MI=$(az identity show -g msaiv2_rg -n msai-gh-oidc --query principalId -o tsv)
az role assignment list \
  --assignee-object-id "$GH_OIDC_MI" \
  --scope /subscriptions/68067b9b-943f-4461-8cb5-2bc97cbc462d/resourceGroups/msaiv2_rg/providers/Microsoft.Network/networkSecurityGroups/msai-nsg \
  --query "[?roleDefinitionName=='Network Contributor']" -o table
```

Expected: 1 row. If `az role assignment list` shows 2 (one from the manual grant, one from Bicep), Bicep's `ghOidcNsgContributorAssignment` uses a `guid()` that doesn't match the manual grant. Idempotency hint: Bicep `guid()` is deterministic, so the second apply should NOT create a duplicate. If you see 2, the manual grant used a different name — delete the older one.

## Failure modes

| Symptom                                                                                   | Diagnosis                                                            | Fix                                                                                           |
| ----------------------------------------------------------------------------------------- | -------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| what-if shows Modify on `Microsoft.Network/networkSecurityGroups/msai-nsg`                | Slice 3 NSG child-resource refactor regressed                        | Inspect main.bicep for inline `securityRules:` property — should be absent                    |
| Apply fails with `AuthorizationFailed` on `Microsoft.Authorization/roleAssignments/write` | Operator lacks `User Access Administrator` or Owner                  | Confirm subscription-level Owner per Slice 1 acceptance                                       |
| `scheduledQueryRules` returns `API version unavailable`                                   | Bicep pinned to a preview API version that's regionally unregistered | Slice 4 alerts.bicep is pinned to `2022-06-15` GA — if you see this error, someone changed it |

## Emergency: backend wedged, need to deploy a fix

When the backend is partial-down (returns 5xx on `/api/v1/live/status`), Slice 4's
`deploy.yml` active-`live_deployments` gate fails closed with
`FAIL_CANNOT_DETERMINE_LIVE_STATE` — refusing to deploy the very fix that would
restore service. Recovery options, in order of preference:

1. **Clear via VM-local CLI (preferred):**

   ```bash
   ssh msaiadmin@platform.marketsignal.ai \
     'cd /opt/msai && sudo docker compose --project-name msai \
        exec -T backend uv run msai live kill-all'
   ```

   `live kill-all` flips the Redis halt flag + signals supervisor without going
   through `/api/v1/live/status`. If backend is fully down, this won't work either
   — go to option 2.

2. **Comment out the gate on a hotfix branch:**
   - Cut a hotfix branch off `main`
   - Comment out the `Refuse if active live_deployments` step in `.github/workflows/deploy.yml`
   - PR + merge + deploy
   - Open follow-up PR to restore the gate IMMEDIATELY after recovery

3. **Last resort — manual deploy on VM:**
   ```bash
   ssh msaiadmin@platform.marketsignal.ai
   cd /opt/msai
   sudo docker compose --project-name msai \
     --env-file /run/msai.env --env-file /run/msai-images.env \
     pull backend frontend
   sudo docker compose --project-name msai \
     --env-file /run/msai.env --env-file /run/msai-images.env \
     up -d --wait backend frontend
   ```
   Skips the migrate one-shot + probes; only safe if no migrations need running.

DO NOT permanently disable the gate. Re-enable in the same hotfix PR.
